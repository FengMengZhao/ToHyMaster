# 功能分析(2017-Q4) & 实体分析(2017-Q4)

## 功能分析

**定义:**

将业务需求转化为软件功能,这些功能可以满足用户所需.

**参与人:**

需求人员,研发经理,架构师,软件设计工程师

**时机:**

功能分析发生在业务分析之后,实体分析之前.

功能分析的依据常常包括:

    - 需求文档
    - 原型图
    - UML图
    - 用户故事

**怎么定义一个功能:**

> 业务分析是梳理要解决的业务问题,而功能分析是提出解决问题的方案

**功能分类:**

- 结构功能(组件化)
    - 主系统
    - 子系统
    - 辅助系统
- 功能本身
    - 主要
    - 基础
    - 辅助

**建模方法:**

- 流程分析法
    - 功能执行顺序,分支,循环的逻辑
    - 落地: 泳道图
- 原型分析法
    - 将结构化的需求进行框架化
- 用例分析法
    - 用户的角度描述软件功能
- 用户故事
    - 作为一个角色,我想要什么功能,以便于实现商业或业务价值

## 实体分析

> 需求分析: 业务分析,功能分析,实体分析

**实体分析定义:**

在业务领域范围内,分析参与其流程的`过程数据`,`自然数据(参与者,地点,对象)`,`角色数据`和`描述数据`是否纳入系统数据及厘清它们之间关系的过程.

- 过程数据: 表证单书
- 自然数据: 参与者,地点,对象
- 角色数据: 将参与者抽象为一种有特性的角色
- 描述数据: 业务领域内设计的规则和概述

**实体分析的目的:**

- 厘清系统业务数据范围及数据间关系
- 原型设计的基础
- 数据构架,数据库表,类代码设计的基础
- 辅助通过数据足迹深入了解业务流程

**实体分析与业务功能分析关系:**

- 业务分析阶段-业务实体分析：
	- 业务分析包含业务实体分析
	- 业务流程活动的输入输出是业务实体的来源
	- 业务实体状态变化是通过业务流程进行转变的
- 功能分析阶段-数据实体分析：
    - 功能分析包含数据实体分析
	- 数据实体体现在功能数据流转过程中

**实体分析 vs 数据库设计:**

- 实体分析 vs 物理结构
    - 实体 vs 表    
    - 实体间关系 vs 表关系
    - 属性 vs 字段
    - 业务约束 vs 约束
    - 关键属性 vs 索引

**实体分析方法:**

- ER图(Entity Relationship Diagram)分析方法
    - 实体
    - 属性
    - 联系(约束关系)
        - 1:1;1:N;N:N
- 实体图分析法
- 领域建模法
    - 红色: 过程数据
    - 绿色: 自然数据
    - 黄色: 角色数据
    - 蓝色: 描述数据
